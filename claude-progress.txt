已查明的 SDK 事实(重要,别重新踩):
1. 帧的 seq **不能** 靠帧本身带:LiveKit 会重写 frame timestamp_us(实测发 0,1000,2000.. 收到的是
   它自己的时钟 1043...);FrameMetadata 是固定 proto。→ 已改成:发布端每帧在 reliable data topic
   "_seq" 上发一条 {seq},订阅端 FIFO 1:1 弹出配对(transport_livekit.py 已实现)。帧用 RGB24 直传。
2. SDK 字段实测:DataPacket(data,kind,participant,topic);VideoFrameEvent(frame,timestamp_us,rotation,
   metadata);capture_frame(frame,*,timestamp_us,rotation,metadata);publish_data(payload,*,reliable,
   destination_identities,topic);VideoBufferType 有 RGB24/RGBA/I420...;VideoFrame.convert(type)。
3. **一个进程里开两个 LiveKit Room、跑在两个 event loop 会冲突**(livekit FFI client 绑单 loop)→
   必须把 daemon 和 controller 当**两个独立进程**测(真实部署本来就是两进程,无所谓)。

当前现象(daemon 作为独立子进程跑):
- daemon 端(publisher)正常(日志有 WATCHDOG 触发,无 "engine is closed");
- controller 端(subscriber,主进程)join 上了 room,但 get_observation() **TimeoutError**
  —— assemble() 一直拿不到“同 seq 的 state+frame”对。
